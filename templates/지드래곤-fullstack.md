## 실행 순서

### Step 0: 파일 설계 계획 (필수)
코딩 전 파일의 책임과 구조를 먼저 설계한다.
```
파일 설계 체크리스트:
1. [파일 목록] 생성/수정할 파일을 모두 나열
2. [책임 정의] 각 파일의 "변경 이유"가 하나인가? (SRP)
3. [기존 파일 확인] 현재 구조 파악
4. [분리 판단] 응집도 기준으로 판단
5. [의존성 맵] 파일 간 관계 미리 설계
```

### Step 1: 요구사항 분석
1. 기능 범위와 영향 레이어 파악
2. 유사 기존 기능의 구현 패턴 참조
3. DB 스키마 변경 필요 여부 판단

### Step 2: DB 스키마 (필요 시)
1. 기존 스키마 확인
2. 마이그레이션 SQL 작성 (UP + DOWN)
3. IF NOT EXISTS / IF EXISTS로 멱등성 보장
4. 적용 및 검증

### Step 3: API/Backend 구현
1. 기존 라우트 패턴 참조
2. 컨트롤러/서비스 레이어 구현
3. 에러 핸들링 및 유효성 검증
4. 타입 정의

### Step 4: Frontend 구현
1. 기존 컴포넌트 패턴 참조
2. UI 컴포넌트 구현
3. API 연동
4. 에러/로딩 상태 처리

### Step 5: 검증
1. 타입 체크 통과
2. 빌드 성공 확인
3. 기능 동작 테스트
4. 엣지 케이스 확인

### Step 6: 완료 (CHANGELOG + Git)
1. CHANGELOG.md 업데이트
2. Git 커밋 (의미 있는 메시지)
